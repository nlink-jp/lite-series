# セキュリティコードレビュー 対応記録

本ドキュメントは lite-series のセキュリティコードレビューで検出された指摘事項、
各指摘への対応方針、および**意図的に対応しないと判断した項目の理由**を記録します。

---

## 実施した修正

### [High] lite-eml — 再帰的 MIME 解析に深度制限なし

**リスク:** 深くネストされた `multipart/*` パーツを含む細工メールによって
コールスタックが枯渇し、プロセスがクラッシュする可能性がある。

**修正内容:** `internal/parser/body.go` に定数 `maxMIMEDepth = 10` を追加。
`parseBody` → `parseMultipart` → `processRawPart` の再帰呼び出しに `depth int`
パラメータを通し、深度が 10 以上のパーツは再帰せず空の結果を返す。

### [Medium] lite-eml — ボディパーツのメモリ上限なし

**リスク:** 非常に大きな MIME ボディパーツを含む細工メールによって
プロセスのメモリが枯渇する可能性がある。

**修正内容:** `internal/parser/body.go` に定数 `maxPartSize = 25 MiB` を追加。
`decodeTransfer` 内で `io.LimitReader(r, maxPartSize)` を使用し、
転送エンコーディングの種類に関わらず 1 パーツあたりの読み取り量を 25 MiB に制限する。

### [Medium] lite-switch — 設定ファイルのパーミッションチェックなし

**リスク:** 設定ファイルが world-readable の場合、共有システム上の他ユーザーに
API キーが漏洩する可能性がある。

**修正内容:** `internal/config/config.go` の `loadSystem` で `toml.DecodeFile`
の前に `os.Stat` を呼び出し、ファイルパーミッションを確認するように変更。
グループ/その他ビットが立っている場合 (`perm & 0077 != 0`)、
`chmod 600` を促す警告を stderr に出力する。

### [Medium] lite-rag — 設定ファイルのパーミッションチェックなし

**リスク:** lite-switch と同様。

**修正内容:** `internal/config/config.go` の `Load` に同様のチェックを追加。

### [Medium] lite-msg — MAPI ストリーム読み込みのメモリ上限なし

**リスク:** 細工された MSG ファイルに意図的に巨大な MAPI ストリームを含めることで、
`io.ReadAll` によるメモリ枯渇攻撃が可能。

**修正内容:** `internal/parser/cfb.go` に定数 `maxStreamSize = 50 MiB` を追加。
`loadDocument` 内の `io.ReadAll(entry)` を
`io.ReadAll(io.LimitReader(entry, maxStreamSize))` に変更し、
1 ストリームあたりの読み取り量を 50 MiB に制限する。

なお、深度制限は不要。OLE2/MAPI 形式はパーサー上フラット（root / recipients / attachments
の 3 層のみ）であり、再帰的なネストは発生しない。

### [Medium] lite-llm — API キーを HTTP で送信しても警告なし

**リスク:** `base_url` が `http://` で始まり `api_key` が空でない場合、
キーが平文でネットワークに流れる。

**修正内容:** `internal/client/client.go` の `client.New` で
`strings.HasPrefix(cfg.Endpoint, "http://") && cfg.APIKey != ""` を確認し、
警告を stderr に出力するように変更。
ローカルエンドポイント（LM Studio など）での意図的な HTTP 利用を妨げないため、
リクエスト自体はブロックしない。

## 意図的に対応しないと判断した項目

### [High] lite-llm — `--file` によるパストラバーサル

**指摘内容:** `--file` フラグが任意のファイルパスを受け付ける。

**対応しない理由:**
`--file` はプロセスを実行しているユーザー自身が指定する CLI 引数である。
ローカル CLI ツールでは、ユーザーはすでに OS レベルで自分のファイルシステム
全体へのアクセス権を持っている。読み取り可能なパスを制限しても
セキュリティ上の意味はなく、`/tmp` や作業ディレクトリ外のファイルを読む
という正当なユースケースを壊す。

パストラバーサル制限が有効なのは、**サーバーサイドのコード**が
信頼できない入力をファイルパスとして解決する場合に限られる。
将来、パスが外部から供給されるモード（例: リモートから取得したバッチファイル）
が追加された場合は、その時点で改めて検討する。

### [High] lite-rag — `--db` およびインデクサーパスによるパストラバーサル

**指摘内容:** `--db` フラグやインデクサーのディレクトリ引数が任意のパスを受け付ける。

**対応しない理由:** lite-llm の `--file` と同じ理由。

### [High] lite-rag — Web サーバー `/api/ask` に認証なし

**指摘内容:** `lite-rag serve` が起動する HTTP サーバーは `/api/ask` を
認証なしで公開しており、ネットワーク上の任意のプロセスから
ナレッジベースを参照できる。

**対応しない理由（現時点）:**
サーバーのデフォルトバインドアドレスは `127.0.0.1:8080`（ループバックのみ）。
他ホストからのリクエストはネットワーク層でブロックされる。

認証を追加するには、トークンの生成・配布・ローテーションの仕組みが必要となり、
これは単純なセキュリティパッチではなく機能追加に相当する。

**現時点の緩和策:** 非ループバックインターフェースへ公開する必要がある場合は、
nginx や Caddy などのリバースプロキシ + HTTP Basic 認証を前段に置くこと。
将来のリリースで `--token` フラグの追加を検討する。

**重要:** デフォルトバインドアドレスを `0.0.0.0` に変更する場合は、
その前に必ず認証機能を実装すること。

### [High] lite-switch — `--config` / `--switches` のパス検証が不十分

**指摘内容:** これらのフラグが任意のパスを正規化・検証なしで受け付ける。

**対応しない理由:** lite-llm の `--file` と同じ理由。ローカル CLI ツールでは
ユーザーが自分でパスを指定するため、制限は不適切。
