# セキュリティパッチ作業手順

本ドキュメントは、lite-series においてセキュリティ上の指摘事項が発生した際の
エンドツーエンドの対応手順を定義します。深刻度に関わらず、すべてのセキュリティ修正でこの手順に従うこと。

---

## 1. トリアージ

コードを書く前に、各指摘事項の対応方針を以下の 3 つから決定する。

| 方針 | 意味 |
|------|------|
| **修正する** | 今サイクルでコード変更を実施する。 |
| **対応しない** | このコードベースにはリスクモデルが当てはまらない（理由を明記する）。 |
| **保留** | リスクは有効だが、修正には大きな機能追加が必要。現状の緩和策と「対応が必須になる条件」を記録する。 |

実装を始める前に、**すべての方針（対応しないものも含む）を**
`docs/security/code-review-findings.md` に記録すること。
→ [指摘事項の記録](#4-指摘事項の記録) 参照。

---

## 2. 影響範囲の確認

指摘事項が複数のプロジェクトに共通して存在していないか確認する。
セキュリティ問題は根本原因を共有することが多い（例: `io.ReadAll` の無制限読み込み、
パーミッションチェックの欠如など）。1 プロジェクトずつ対応するのではなく、
同一サイクルで影響を受けるすべてのプロジェクトを一括で修正する。

---

## 3. 実装

- 1 プロジェクトにつき 1 コミット。セキュリティ修正のみにスコープを限定する。
  無関係なリファクタリングや機能追加を同じコミットに混ぜない。
- コミットメッセージの書式:

  ```
  security: <命令形の短い説明>

  <任意のボディ — リスクと修正内容の説明>
  ```

- 修正後、既存テストがすべて通ることを確認する（`go test ./...`）。
- 実用的な範囲でテストを追加する（深度制限による弾き、サイズ制限による切り捨てなど）。

---

## 4. 指摘事項の記録

`docs/security/code-review-findings.md`（および `docs/ja/security/`）を更新する。

**修正済み項目**
- リスクの内容、変更内容、追加した定数やガードの説明。

**対応しない項目**
- リスクモデルが当てはまらない理由を明示する。CLI ツールにおける標準的な理由:

  > このフラグはプロセスを実行しているユーザー自身が指定する CLI 引数である。
  > ユーザーはすでに OS レベルでリソースへのアクセス権を持っている。
  > パス・入力の制限が有効なのは、信頼できない入力をサーバーサイドで解決する
  > 場合に限られる。

**保留項目**
- 現状の緩和策と、対応が必須になるトリガー条件を明記する
  （例:「デフォルトバインドアドレスをループバックから 0.0.0.0 に変更する前に必ず実装すること」）。

指摘事項ドキュメントは監査証跡である。セキュリティレビューの全項目がここに記録されていること。

---

## 5. CHANGELOG の更新

影響を受ける各プロジェクトの `CHANGELOG.md` に新バージョンエントリを追加し、
`### Security` セクションを設ける。内部実装ではなく、リスクと対策を平易に記述する。

記述例:

```markdown
## [0.1.1] - 2026-03-27

### Security

- MIME 再帰深度制限（`maxMIMEDepth = 10`）を追加。
  細工された深くネストされたマルチパートメッセージによるスタック枯渇を防ぐ。
- パーツ単位のメモリキャップ（`maxPartSize = 25 MiB`）を追加。
  巨大なボディパーツによるメモリ枯渇を防ぐ。
```

---

## 6. バージョン番号の決定

セキュリティのみのパッチは**パッチバージョン**を上げる（`0.x.Y → 0.x.Y+1`）。

セキュリティ修正に伴い破壊的 API 変更が含まれる場合は、
**マイナーバージョン**を上げる（`0.x.Y → 0.(x+1).0`）。

---

## 7. 各プロジェクトのリリース

影響を受けるプロジェクトそれぞれについて、標準の
[リリースプロセス](../../CONVENTIONS.md#release-process) に従う。
1 プロジェクト内の手順:

1. `make check` — テストが全件通ること。
2. CHANGELOG に `### Security` セクションを追記済みであること。
3. 指摘事項ドキュメント（EN + JA）が更新済みであること。
4. `main` にコミット・プッシュ。
5. `git tag vX.Y.Z` してタグをプッシュ。
6. `make build-all`（CGO プロジェクトは `make dist`）。
7. GitHub Release を作成し、全プラットフォームのアーカイブをアセットとしてアップロード。

---

## 8. サブモジュールポインタの更新

影響を受けるすべてのプロジェクトのリリース後、
`lite-series` アンブレラリポジトリを更新する:

```bash
git submodule update --remote <project1> <project2> ...
git add <project1> <project2> ...
git commit -m "chore: update submodule pointers to security release commits"
git push
```

**見落としやすいステップ。スキップしないこと。**
